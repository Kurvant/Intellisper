**Org memory** is your workspace's shared knowledge. Unlike personal memory, it isn't tied to whoever
wrote it — it **belongs to the organisation**. Every member of the workspace can **read and curate** it:
add facts, edit them, or remove them.

This is the place for knowledge the whole team benefits from: "our support email is help@example.com,"
"we refer to customers as 'members'," "the fiscal year starts in April."

## Who can see and change it

Everyone in the workspace. Org memory is deliberately **not** scoped to individuals — any member can read
every org fact and any member can curate it. There's no private-vs-shared distinction here: an org fact
is, by definition, shared with the workspace.

:::info This is different from sharing a personal fact
Putting knowledge in **org memory** is a direct choice to make it team-wide. It's separate from the
[three-condition gate](/memory/privacy-and-sharing) that governs whether your *personal* facts can become
visible to an admin. Org memory is shared by construction; personal sharing is tightly gated.
:::

## How it's used

- The **Studio Copilot** recalls relevant org facts (alongside your personal ones) to answer in-app.
- **Flow steps** can read and write org memory through the engine memory surface — so an automation can
  draw on shared team knowledge.

This makes org memory the natural home for facts you want your automations and your team's Copilot to act
on consistently.

## Curating it

Because org memory is shared, curation is a team responsibility. Any member can add a fact, correct one,
or remove one that's out of date. Keep it accurate — everyone (and the Copilot, and your flows) relies on
it.
