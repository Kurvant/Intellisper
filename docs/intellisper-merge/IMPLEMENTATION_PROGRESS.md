- 2026-07-14 вЂ” **Phase 9b вЂ” AI provider-key + credit unification.** The browser agent now draws AI from the SAME platform provider/credit layer as the automation side, so all AI spend is one governed pool.
  - **Decisive finding:** there is **no manual credit-debit primitive** in this codebase. Credits are consumed *exclusively* by routing the model call THROUGH the platform's managed OpenRouter key (OpenRouter meters real usage against the key's spend limit; the app only ever READS usage or RAISES the limit). So a "read billedTokens в†’ manually decrement" design would be a second, parallel accounting system that double-counts. Routing through the managed key was therefore mandatory, not optional.
  - **Key resolver** (`model-provider/model-key-resolver.ts`): per-platform MANAGED (enriched OpenRouter key, gated on `aiCreditsEnabled`) vs ENV (`BROWSER_AGENT_*` keys вЂ” the CE/self-host path, unchanged so nothing that works today breaks). Fail-safe (any error в†’ env, never blocks a turn) and per-platform TTL-cached so the hot path is a memory hit, not a per-turn DB read + decrypt.
  - **Caching preserved (verified in the provider's source):** `@openrouter/ai-sdk-provider`'s `getCacheControl` reads BOTH the openrouter AND the **anthropic** `cacheControl` convention вЂ” so the facade's existing prompt-caching breakpoint is honoured unchanged, and `promptTokensDetails.cachedTokens` flows back so `billedTokens` stays exact.
